01 — Fuente de datos

La consulta real contra SAP.

El código de color por sí solo no es legible para el cliente final. SAP expone su traducción a nombre comercial en la tabla de usuario @ARGNS_COLOR.

02 — Regla de mapeo

Dos niveles: familia y nombre comercial.

El código no es arbitrario — el primer dígito (o prefijo, en códigos alfanuméricos) determina la familia de color. Dentro de cada familia, el código completo resuelve el nombre comercial específico que ve el cliente final.

000Blanco
0xxAmarillo
1xxRojo
2xxNaranja
3xxRosa
4xxMorado
5xxAzul
6xxVerde
7xxMarrón
8xxNegro / Gris

Resolución en dos pasos, con ejemplos reales:

1

Familia por rango del código

730 → primer dígito 7 → familia Marrón

2

Nombre comercial por código exacto

730 → fila exacta en @ARGNS_COLOR → "Café"

Por qué importan los dos niveles

El nombre comercial (paso 2) es lo único que se envía a Shopify como opción de variante. La familia (paso 1) no se expone al cliente — sirve como agrupador interno opcional para QA y para detectar códigos fuera de rango durante las pruebas.

03 — Calidad de datos

3 códigos con nombre en conflicto.

Sobre 1,153 filas verificadas en el archivo de pantones de referencia, 3 códigos numéricos tienen más de un nombre comercial asociado. El resto de las repeticiones encontradas son filas idénticas (mismo código, mismo nombre), sin conflicto real.

CodeNombres encontradosRegla de resolución
760"Café" / "Kaki"Primer registro por ID
780"Café" / "Kaki"Primer registro por ID
790"Beige" / "Kaki"Primer registro por ID
!

Regla de resolución confirmada: ante un código con más de un nombre en @ARGNS_COLOR, el lookup toma el primer registro encontrado (menor ID / orden de inserción) y descarta los siguientes. No se requiere intervención manual para que el addon siga funcionando — pero estos 3 códigos quedan marcados para que Van Heusen los depure en el origen cuando pueda.

04 — Arquitectura de armado del padre

Solo lo que tiene stock real.

El padre se construye con SAP como única fuente de verdad — la API de Shopify nunca decide qué variantes existen, solo recibe lo que el addon ya resolvió.

SAP B1 Data pura ItemCode, Modelo, Color, Talla, Precio (Lista 02), Stock (OITW)
→
Addon Filtrado + traducción Omite color sin stock · resuelve nombre via @ARGNS_COLOR
→
Shopify Padre + variantes Modelo = producto · Color×Talla = variante · SKU = ItemCode
Regla 1

Imágenes por Modelo + Color

La búsqueda en Amazon S3 usa el prefijo Modelo+Color — no solo Modelo. Cada color tiene su propia sesión de fotos; agrupar solo por Modelo mostraría la imagen incorrecta en colores que no son el principal.

Regla 2

Omisión por color sin stock

Si un color no tiene stock disponible en ninguna de sus tallas, ese color se omite por completo del padre. No se expone como variante deshabilitada — simplemente no existe en el payload enviado a Shopify.

Regla 3

Tallas asimétricas por color

No todos los colores tienen las mismas tallas. El padre debe exponer únicamente las combinaciones Color×Talla que existen y tienen stock — nunca generar el producto cartesiano completo de todos los colores por todas las tallas.

Regla 4

Degradación sin traducción

Si el código de color no tiene fila en @ARGNS_COLOR, la variante se sincroniza igual pero sin Color como opción visible — queda solo con Talla. No se bloquea la sincronización completa por un color sin traducir.

05 — Cómo se toma en SAP

Qué se lee, de dónde, y con qué llave.

Todo el ciclo arranca con una lectura completa por Modelo — nunca por variante aislada. Leer una sola variante sin su grupo completo es la causa raíz del bug original (productos duplicados) que motivó este rediseño.

Tabla SAPCampoRol en el ciclo
OITMItemCodeSKU exacto de la variante en Shopify (ver sección 08)
OITMU_ARGNS_MODIdentifica el Modelo → es la clave del producto padre
OITMU_ARGNS_COLCódigo de color → resuelto a nombre vía @ARGNS_COLOR (sección 2)
OITMU_argns_sizeTalla → segunda opción de variante
OITMU_ShopifyStatusBandera de sincronización pendiente/hecho (significado completo: ver documento de análisis funcional, pendiente #02)
ITM1Lista de Precios 02Precio de venta de la variante
OITWOnHandStock físico, filtrado WhsCode = 'A01'
OITWIsCommitedStock comprometido — se resta del físico (ver fórmula abajo)
@ARGNS_COLORCode, NameTraducción código→nombre comercial de color (sección 1-2)
SQL · Lectura completa de un Modelo (vía Service Layer / SQL directo)
SELECT
    T0.ItemCode,
    T0.U_ARGNS_MOD   AS Modelo,
    T0.U_ARGNS_COL   AS CodigoColor,
    T0.U_argns_size  AS Talla,
    T0.U_ShopifyStatus,
    T1.Price         AS PrecioLista02,
    T2.OnHand        AS StockFisico,
    T2.IsCommited    AS StockComprometido,
    (T2.OnHand - T2.IsCommited) AS Disponible
FROM OITM T0
LEFT JOIN ITM1 T1 ON T1.ItemCode = T0.ItemCode AND T1.PriceList = 2
LEFT JOIN OITW T2 ON T2.ItemCode = T0.ItemCode AND T2.WhsCode = 'A01'
WHERE T0.U_ARGNS_MOD = @modelo
i

Por qué se lee por Modelo completo y no por ItemCode aislado: el addon necesita ver todas las variantes (todos los colores y tallas) de un Modelo en la misma operación de lectura, antes de decidir qué color sobrevive (Regla 2, stock total) y qué tallas expone cada color (Regla 3). Leer ItemCode por ItemCode de forma aislada no permite aplicar ninguna de las dos reglas — es la causa estructural del bug original (productos duplicados) que originó este rediseño completo.

06 — Reconciliación contra Shopify

¿Crear producto nuevo, o actualizar uno existente?

Como SAP no guarda el Shopify Product ID (el padre se identifica por Modelo, no por un ID externo cacheado), cada corrida debe preguntarle a Shopify si el producto de ese Modelo ya existe, antes de decidir.

!

Decisión técnica recomendada, pendiente de validar con el equipo: ya que no hay Shopify Product ID cacheado en SAP, el Modelo debe quedar guardado en Shopify como un metafield propio del producto (ej. custom.sap_modelo), y la reconciliación se hace consultando ese metafield — no por handle ni por título, que pueden cambiar por edición manual en Shopify sin que SAP se entere.

1

Buscar producto existente por metafield de Modelo

Query GraphQL products(query: "metafield:custom.sap_modelo:{modelo}") — si devuelve resultado, el padre ya existe y se sigue el camino de actualización.

2

Si no existe — crear producto padre completo

Se crea el producto con todas las variantes que sobrevivieron las Reglas 2 y 3 en la misma operación (ver sección 07). El metafield de Modelo se escribe en esta misma llamada.

3

Si existe — diferenciar variantes nuevas vs. existentes

Comparar el conjunto de SKU (ItemCode) que SAP entrega contra los SKU que Shopify ya tiene en ese producto. Los que están en SAP pero no en Shopify → agregar. Los que están en ambos → actualizar precio/stock. Los que están en Shopify pero ya no en SAP con stock → evaluar regla de Draft/Archived (pendiente #03 del análisis funcional).

07 — Cómo se arma para Shopify

Un producto, todas sus variantes, una sola llamada.

Mecanismo de API recomendado para la creación inicial del padre completo — pendiente de confirmar contra la versión de API que ya use el resto del addon.

i

Recomendación técnica: usar la mutación GraphQL productSet (Admin API 2024-01+), que crea o reemplaza un producto completo — título, opciones, variantes y precios — en una sola llamada idempotente. Es preferible sobre encadenar productCreate + productVariantsBulkCreate por separado, porque reduce el riesgo de que el producto quede creado sin variantes si la segunda llamada falla a mitad de la sincronización.

GraphQL · productSet — payload de ejemplo (Modelo BF5CACC06)
mutation {
  productSet(input: {
    title: "BF5CACC06 — Blazer Dress Solido"
    metafields: [
      { namespace: "custom", key: "sap_modelo", value: "BF5CACC06", type: "single_line_text_field" }
    ]
    productOptions: [
      { name: "Color", values: [{ name: "Navy" }, { name: "Café" }] },
      { name: "Talla", values: [{ name: "S" }, { name: "M" }, { name: "L" }] }
    ]
    variants: [
      {
        sku: "BF5CACC06-500-S"
        optionValues: [{ optionName: "Color", name: "Navy" }, { optionName: "Talla", name: "S" }]
        price: "716.00"
        inventoryQuantities: [{ availableQuantity: 2, locationId: "..." }]
      },
      {
        sku: "BF5CACC06-500-M"
        optionValues: [{ optionName: "Color", name: "Navy" }, { optionName: "Talla", name: "M" }]
        price: "716.00"
        inventoryQuantities: [{ availableQuantity: 3, locationId: "..." }]
      }
      // ... una entrada por cada combinación Color×Talla que sobrevivió Reglas 2 y 3
    ]
  }) {
    product { id }
    userErrors { field message }
  }
}
Creación

Producto nuevo (no existía el Modelo)

productSet con todas las variantes sobrevivientes en el mismo arreglo. El metafield sap_modelo se escribe en esta misma llamada — sin eso, la reconciliación de la sección 06 no tiene nada contra qué buscar la próxima vez.

Actualización

Producto existente — variantes nuevas

productVariantsBulkCreate únicamente para los SKU que SAP entrega y Shopify aún no tiene. No se vuelve a enviar el producto completo — solo las variantes faltantes.

Actualización

Producto existente — variantes que ya existen

productVariantsBulkUpdate (precio) + mutación de inventorySetQuantities (stock) por separado — son dos recursos distintos en la API de Shopify, no se actualizan en la misma llamada.

Pendiente

Ubicación de inventario (locationId)

El payload de ejemplo asume una sola ubicación de inventario en Shopify mapeada al almacén A01 de SAP. Confirmar con Gilberto si Van Heusen tiene más de una ubicación activa en Shopify antes de fijar este valor como constante.

08 — El control crítico: SKU = ItemCode

La única llave que importa después de creado el producto.

Una vez que el padre y sus variantes existen en Shopify, todo el resto del ciclo de vida — actualizar precio, actualizar stock — depende de una sola cosa: que el campo sku de la variante en Shopify sea idéntico, carácter por carácter, al ItemCode de SAP.

SAP ItemCode Ej. BF5CACC06-500-S — generado por SAP, nunca por el addon
=
Shopify variant.sku Debe coincidir exactamente — es la llave de cruce para siempre

Por qué este control es más importante que cualquier otra regla del documento:

Riesgo

Shopify autogenera sufijo si el SKU llega vacío o duplicado

Si el payload de creación no envía sku explícito en cada variante, o envía el mismo valor dos veces en la misma llamada, Shopify le agrega un sufijo automático (-1, -2...). Esto ya se documentó como causa de bug en el análisis funcional (pendiente #04) — el control de este documento es la forma de prevenirlo desde el origen, no solo detectarlo después.

Validación obligatoria

Unicidad garantizada por construcción, no por casualidad

Como el ItemCode ya es Modelo-Color-Talla y SAP no permite dos artículos con el mismo ItemCode, la unicidad del SKU dentro de un mismo producto está garantizada si y solo si el payload nunca transforma, recorta o normaliza el ItemCode antes de enviarlo como SKU.

Ciclo de actualización

Precio y stock se actualizan SIEMPRE por SKU, nunca por nombre de variante

El lookup para actualizar precio/stock (Proceso 2) debe buscar la variante en Shopify por sku = ItemCode exacto — nunca por la combinación de nombres de Color/Talla, que puede ambigüarse si dos colores resuelven al mismo nombre comercial (ver sección 3, códigos 760/780/790).

Caso límite

¿Qué pasa si alguien edita el SKU manualmente en Shopify?

Pendiente de definir con el equipo: si un usuario de Shopify cambia el SKU a mano, el siguiente ciclo de sincronización no va a encontrar coincidencia y probablemente cree una variante duplicada. Recomendado: agregar una validación de integridad que alerte (no que corrija solo) si el conteo de variantes esperadas vs. encontradas por SKU no coincide.

09 — Contrato de implementación

El ciclo completo, paso a paso.

Desde la lectura en SAP hasta el envío a Shopify — el orden de estos pasos no es intercambiable. Cada paso depende del resultado del anterior.

Pseudocódigo · Ciclo completo SAP → Shopify por Modelo 
PASO 1 — Leer todas las variantes del modelo desde SAP (Service Layer)
  variantes = SELECT ItemCode, U_ARGNS_MOD, U_ARGNS_COL, U_argns_size,
                     U_ShopifyStatus, Precio (ITM1, Lista 02),
                     OnHand, IsCommited, (OnHand - IsCommited) AS Disponible
              FROM OITM + ITM1 + OITW
              WHERE U_ARGNS_MOD = {modelo} AND OITW.WhsCode = 'A01'

PASO 2 — Agrupar por color y calcular stock total del color
  Para cada color distinto en variantes:
      stock_total_color = SUMA(Disponible) de todas las tallas de ese color
      SI stock_total_color == 0:
          DESCARTAR este color por completo (Regla 2)
          continuar con el siguiente color

PASO 3 — Resolver nombre comercial del color (lookup en vivo)
  Para cada color que sobrevivió el Paso 2:
      fila = SELECT TOP 1 Name FROM @ARGNS_COLOR
             WHERE Code = {codigo_color}
             ORDER BY {ID o campo de orden de inserción}
      SI fila existe:  nombre_color = fila.Name
      SINO:            nombre_color = NULL   (Regla 4: sin opción Color visible)

PASO 4 — Resolver imagen del color (Amazon S3)
  prefijo_busqueda = Modelo + "-" + codigo_color     (Regla 1)
  imagen = buscar en bucket S3 por prefijo_busqueda

PASO 5 — Reconciliar contra Shopify (sección 06)
  resultado = buscar producto en Shopify por metafield custom.sap_modelo = {modelo}
  SI resultado es vacío:
      ir a PASO 6A (crear producto nuevo)
  SINO:
      ir a PASO 6B (actualizar producto existente)

PASO 6A — Crear producto padre completo (sección 07)
  armar payload productSet con:
      title, metafield sap_modelo = {modelo}
      productOptions = [Color, Talla]
      variants = una entrada por cada combinación Color×Talla
                 que sobrevivió Pasos 2-4, con:
                     sku = ItemCode EXACTO (sección 08 — sin transformar)
                     price = Precio
                     inventoryQuantities = Disponible
  enviar productSet a Shopify

PASO 6B — Actualizar producto existente
  sku_sap = conjunto de ItemCode resueltos en Pasos 2-4
  sku_shopify = conjunto de SKU que el producto ya tiene en Shopify
  nuevos = sku_sap - sku_shopify        → productVariantsBulkCreate
  existentes = sku_sap ∩ sku_shopify    → productVariantsBulkUpdate (precio)
                                         → inventorySetQuantities (stock)
  faltantes = sku_shopify - sku_sap     → evaluar regla Draft/Archived
                                           (pendiente #03, análisis funcional)

PASO 7 — Confirmar en SAP
  SI Pasos 6A/6B respondieron sin userErrors:
      actualizar U_ShopifyStatus de cada ItemCode procesado
  SINO:
      registrar error para reintento — no marcar como sincronizado

10 — Casos de prueba

Para validar antes de producción.

CasoEntradaResultado esperado
TC-01 Color con código en @ARGNS_COLOR y stock > 0 en al menos una talla Color aparece con nombre comercial correcto y solo sus tallas con stock
TC-02 Color con stock = 0 en todas sus tallas Color completamente omitido del payload (Regla 2)
TC-03 Código de color sin fila en @ARGNS_COLOR Variante sincronizada sin opción Color, solo Talla (Regla 4)
TC-04 Código 760, 780 o 790 (nombre conflictivo) Se usa el primer registro por ID; no se bloquea ni se duplica
TC-05 Modelo con colores que tienen distintas tallas disponibles entre sí Cada color expone solo sus propias tallas — sin producto cartesiano completo
TC-06 Búsqueda de imagen para un color sin foto propia en S3 Documentar como pendiente — no cubierto por esta especificación; confirmar fallback con Gilberto
